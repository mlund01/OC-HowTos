RESTful API Intro

What is a RESTful API?

A RESTful (Representational State Transfer) API is an increasingly popular software architecture that enables scalable, maintainable, and performant web services. A RESTful API exposes its underlying data model the same way your browser makes a request to a server to receive a web page, through the HTTP protocol. What this means is that, 1. You don't have to incorporate any additional technology to process your requests, and 2. It's a lot easier! Additionally, RESTful API allows the client to interact with it in the form of standard data objects and markup (XML, JSON, HTML), which are conceptually different than the representations of the data stored on the server. This means you don't have to worry about dealing with advanced data manipulation or conversions to send and receive data, so you can focus on building applications.

What makes a RESTful API "RESTful?"

There are a few key factors that make a RESTful API "RESTful",

  1. Client-Server Separation
    • This means the client never has to inherently concern itself with data storage, that's all handled on the server side through the API
  2. Stateless
    • A RESTful API never concerns itself with the state of the client, and every request from the client contains everything necessary to handle the request
  3. Cacheable
    • A RESTful API will in some way state that it is cacheable or not, ensuring that data isn't stale or misused in your application
  4. Layered System
  5. Code on demand (optional)
  6. Uniform Interface
    • A Uniform Interface ensures that you will never have to worry about the API fundamentally changing the way in which it works, meaning you can be rest assured that your app won't one day crash because of an API update

RESTful API Methods

Every RESTful API has a standard set of Methods that allow you to interact with the data model. They all serve a different purpose, but together offer an exhaustive tool-set to interact with the data model. The Methods used on OrderCloud 3.0 are,

  1. GET
  2. POST
  3. PUT
  4. PATCH
  5. DELETE

GET is one of the most commonly used Methods, and allows you to do exactly as the name suggests. GET is what you will use to access any data objects that are already stored on the server. This Method is read-only, meaning you will only be able to access information stored on the server, never manipulate it.

POST is also a very commonly used Method in OrderCloud 3.0, and allows you to send new objects to be created and stored on the server. With a POST Method call, you must also send an object as part of the request, like so,

PUT is a utility method in RESTful API's. It allows you to create OR update an object stored on the server. With a PUT method call, you must also send an object in the request. If the ID of the object is not found on the server, a new object will be created with that ID. If the ID of the object is found, a PUT method will update the object with the new object you send in the request. Note that if you are using PUT to update a pre-existing object, you must send a full object in your request. This means you cannot send it an object with only the updates you wish to make.

PATCH is similar to a PUT in that it updates a pre-existing object already stored on the server. The difference is that with a PATCH method call, you only have to send the changes in the request object vs. the entire object. This works great for when you are only updating a small piece of the object you are targeting.

DELETE will remove an entire object from server. You will not have to send an object with a DELETE method call, as the information of the target object you are attempting to delete will either be in the parameters, or in the URL params.

OrderCloud 3.0 Console

To the right, you will see three different containers. The large container on the left is the OrderCloud 3.0 Console. This is a fully function HTTP protocol-based console for you to send real API requests, much like you would see in Google's PostMan API Console or Fiddler.

In the bottom right corner, you will see your response console. This is where any response data from your API request will end up, and will also be the first place you'll want to look if you run into any errors, as the API has robust error messaging functionality.

And finally, in the upper right hand corner, we have provided you with a section for helpful hints to make your life a little bit easier when it comes to processing your API requests, focusing on whatever class you are currently taking.

We have provided you with an example request in the console that will give you a taste of what the console will look and feel like as you move on through the tutorials. Go ahead and review the Method, URL, and Parameters at the top of the OrderCloud 3.0 Console. The Method, Location, and Content-Type/Authorization Params will be a part of every request you make. When you are ready, go ahead and click submit, and you will see a sample response in the response console.

The OrderCloud 3.0 Console will be available to you in every class you take. You have the option to pre-fill the method, url, and paramaters in every course (change the settings in the upper right hand corner of the console container). From now on (after this page), you will be able to make any request you want, in any course, at any time.
